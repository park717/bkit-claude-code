📚 Living Context System (4-Layer)
Self-Healing 수정 전 자동 로딩되는 컨텍스트
📋
Layer 1: Scenario Matrix
각 모듈의 시나리오 정의: 입력/출력/경계값/에러케이스
why (왜 이렇게 만들었는지) + constraint (건드리면 안 되는 것)
docs/02-design/scenarios/*.yaml
PDCA Design에서 자동 생성
🔒
Layer 2: Invariants Registry
핵심 비즈니스 룰 불변 조건 (예: "잔액 ≥ 0", "주문 역행 불가")
critical 위반 시 수정 거부 → 사람 에스컬레이션
.bkit/invariants.yaml
사람이 핵심만 작성 (10~20개)
🕸️
Layer 3: Impact Map
모듈 간 의존관계 + blast radius 자동 산정
수정 파일의 영향받는 파일 연쇄 검증
.bkit/impact-map.yaml
code-analyzer 정적 분석 자동 생성
🧠
Layer 4: Incident Memory
과거 장애 기록 + 교훈 + anti-pattern
같은 영역 동일 실수 반복 방지
.bkit/incident-memory.yaml
Self-Healing 실행 시 자동 누적
🔄 Context Loading & Verification Flow
① 에러 위치 파일 식별 (Stack Trace 파싱)
↓
② Context Loader — 4-Layer 자동 로딩
scenarios/*.yaml → invariants.yaml → impact-map.yaml
→ incident-memory.yaml → PDCA Plan/Design 문서
↓
③ Claude Code — 컨텍스트 포함 수정 생성
↓
④ Scenario Runner — 4중 검증
✓ 시나리오 매트릭스 전체 통과?
✓ 불변 조건 위반 없음? (critical = 수정 거부)
✓ blast radius 내 다른 파일 안전?
✓ 과거 anti-pattern 반복 안 함?
↓
⑤ Pass → Auto PR / Fail → 재시도(max 5) / 5회 실패 → 에스컬레이션

🔄 bkit PDCA Integration Map
Plan → 인프라 요구사항 + Invariants 초안 작성
Design → 아키텍처 설계 + Scenario Matrix 자동 생성
Do → 코드 구현 + Impact Map 자동 생성
Deploy → dev → staging → prod (서브 상태 머신)
Check → gap-detector + 운영 메트릭 3축 (Obs)
Act/Heal → Living Context 로딩 → 컨텍스트 인지형 수정
Report → 운영 메트릭 포함 + Incident Memory 자동 누적
⚡ Automation Levels (L0-L6)
L0 Manual · L1 Guide · L2 Semi-Auto ← default · L3 Auto · L4 Full Auto
L5 Deploy Auto — CI/CD + 환경 승격 자동 NEW
L6 Self-Heal Auto — 에러→수정→PR 자동, 리뷰만 사람 NEW
🧑‍💻 Human Intervention Points (6개) — 자동화 범위를 최대화하면서도 절대 우회 불가한 사람 개입 지점
① 코드 작성 & 커밋 — Developer가 코드를 작성하고 git commit/push. 모든 파이프라인의 시작점.
② Grafana 대시보드 관제 — Metrics + Logs + Traces 3축 통합 관제. 알림 임계값 설정.
③ Slack 알림 확인 & 판단 — P1~P4 심각도 판단, 자동 수정 대상 vs 사람 직접 대응 결정.
④ On-Call 긴급 장애 대응 — PagerDuty 에스컬레이션 시 인프라/DB 장애 직접 대응.
⑤ PR Review & Prod Approval — Self-Healing PR 코드 리뷰 + 프로덕션 배포 최종 승인 게이트.
⑥ 에스컬레이션 대응 — Self-Healing 5회 실패 시 #eng-escalation 채널에서 사람이 직접 수정.
